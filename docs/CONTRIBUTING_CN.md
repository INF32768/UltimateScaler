## 🌟 我们的贡献者

我们坚信，每一位为项目付出努力的人都值得被铭记。因此，我们维护着一份 **[贡献者名单](CONTRIBUTORS_CN.md)**。

**你可以通过以下任何一种方式登上这份名单：**
- 提交有效的 **Issue**（对于问题 Issue，审查后不被标记为 `invalid` 或 `duplicate` 即为有效；对于建议 Issue，审查后标记为 `enhancement` 才算有效）。
- 提交的 **Pull Request** 被合并。
- 通过 GitHub Discussions、B站私信或视频评论等社交媒体提供了**关键性的帮助或建议**（由维护者认定）。
- 提供了重要的**非代码贡献**，如文档、翻译、设计等。

我们相信，社区的力量源于每一个微小的贡献。这份名单是我们对所有帮助者最诚挚的致谢。

## 🪲 报告问题

在提交问题 Issue 前，请先搜索是否已有类似问题。

**一个优秀的 Issue 应包含：**
1.  **清晰的标题**：如 `[1.21.1] 启用 BigInteger 重写后，进入世界时崩溃`。
2.  **详细描述**：
    - Minecraft 版本、Mod 版本、Fabric Loader 版本。
    - 复现步骤（尽量简洁）。
    - 预期行为与实际行为。
    - 相关的游戏日志（使用 [gist](https://gist.github.com/) 或类似服务粘贴）。
3.  **（可选）附加信息**：截图、崩溃报告、相关 Mod 列表。

**在提交 Issue 前，请验证：**
- 你已经阅读过配置 GUI 中的**常见问题解答**和对应选项的**描述**，尽量确保你的问题不是预期行为或不予修复的已知问题。
- 目前 Issues 列表中没有与你的问题重复的 Issue。
- 你的问题在给定条件下**可稳定复现**。
- 你使用的是**最新正式版或最新每日构建版**的模组。

## 💡 提议新功能

如果你有任何能增强模组研究能力或用户体验的想法，欢迎提交建议 Issue！

**在提议前请思考：**
- 这个功能是否与模组“探索地形生成边界和距离现象”的核心目标相符？
- 是否有变通方案？你的提议能带来多大提升？
- 这个功能的可行性如何？是否已有相关技术或工具？

**提交时请说明：**
1.  **功能概述**。
2.  **动机**：为什么需要它？它能解决什么现有问题？
3.  **（可选）实现思路**：如果你有技术想法，欢迎一并提出。

_不要被这些规则吓到，大胆地提出你的想法！我们对每一份建议都很重视，并乐意帮助你实现它。_

## 🧑‍💻 贡献代码

我们热烈欢迎代码贡献！以下是快速上手指南：

### 开发环境
1.  **Fork** 本仓库。
2.  克隆你的 Fork：
    ```bash
    git clone https://github.com/<你的用户名>/UltimateScaler.git
    ```
3.  按照 [README](README_CN.md#开始开发) 中的说明配置开发环境。
4.  创建一个新的功能分支：
    ```bash
    git checkout -b feat/your-feature-name
    ```

### 提交规范
我们使用 **Conventional Commits** 规范来保持提交历史的清晰，**描述必须用英语撰写**。
- `feat:` 新功能
- `fix:` Bug 修复
- `docs:` 仅文档更改
- `style:` 不影响代码含义的格式更改（缩进、空格等）
- `refactor:` 既不修复 Bug 也不增加功能的代码优化
- `perf:` 提升性能的代码更改
- `test:` 添加或修改测试
- `chore:` 对构建过程或辅助工具的更改

**示例：**
```bash
git commit -m "feat: add support for offsetting structure generation"
git commit -m "fix(core): prevent crash when noise sampler is null"
```

### 代码风格
- 遵循现有的代码格式（我们主要遵循 Java 通用约定）。
- 为新增的公共类和方法编写清晰的 Javadoc 注释。
- 确保你的更改能通过现有测试（`./gradlew test`）。

### 发起 Pull Request
1.  将你的分支推送到你的 Fork：
    ```bash
    git push origin feat/your-feature-name
    ```
2.  在 GitHub 上发起 Pull Request 到本仓库的 `main` 分支。
3.  在 PR 描述中，清晰地说明你的更改内容、动机，并关联任何相关的 Issue。
4.  等待代码审查。我们可能会请求一些更改或进行讨论。
